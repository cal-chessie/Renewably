# RENEWABLY — AUDIT (HERMES)

**Date:** 2026-07-15
**Audited against:** local clone `COMH/RENEWABLY/website/New Renewably Website/Renewably New Website` (= `cal-chessie/Renewably`)
**Method:** read actual source (contact, agent, chat, middleware, next.config, crm-auth, .env) + regex. NOT a live deploy test.
**Context:** Renewably is a far more mature app than RI — 60 API routes, full CRM, billing, calendar, agent content API. Security posture is generally BETTER than RI (headers in code, gitignored .env, real CRM session validation). But several real issues remain.

## ⚠️ LOCAL ≠ LIVE (same caveat as RI)
The user confirmed disk ≠ deploy earlier. `/api/agent` key finding depends on whether the LIVE env has `AGENT_API_KEY` set — the local `.env` does NOT, so the hardcoded fallback is active on this clone. Verify the live Vercel env before trusting it's fixed.

## 🔴 CRITICAL
1. **`/api/agent` hardcoded fallback key (src/app/api/agent/route.ts line 5).**
   `const AGENT_API_KEY = process.env.AGENT_API_KEY || "change-me-in-production";`
   The local `.env` has NO `AGENT_API_KEY` → the fallback `"change-me-in-production"` is ACTIVE on this clone. Anyone sending `x-agent-api-key: change-me-in-production` can GET/POST/PUT/DELETE all blog/services/testimonials/pages. **Set `AGENT_API_KEY` in live env immediately, or the content API is wide open.** (The auth function compares the header to this value — so the default IS the working key.)

2. **`/api/contact` swallows DB failure, returns 200 (src/app/api/contact/route.ts lines 146-151, 220-223).**
   If Prisma save throws, it's caught, logged as `console.warn`, and the handler STILL returns `{ success: true, message: "Thank you! We'll be in touch..." }`. A lost lead looks like a success. (Mitigation: it returns 503 only if BOTH db+email fail — line 197. So a db-only failure = silent lead loss with 200.) Same class of bug as RI's lead path, but less severe here.

## 🟠 HIGH
3. **Chat route uses `z-ai-web-dev-sdk` (src/app/api/chat/route.ts, 16KB).** Same cold-start / Vercel-build risk as RI #1. Confirm it builds + runs on Vercel hobby; if the package isn't installable/compatible, chat is dead in prod despite looking fine.

4. **`ignoreBuildErrors: true` (next.config.ts line 10).** TypeScript errors are silenced at build. Type bugs ship to production. Should be `false` before launch (there's even a TODO saying so). Risk: latent type errors in 60 routes.

5. **Middleware CRM guard checks cookie PRESENCE only (src/middleware.ts line 35).** `cookieHeader.includes('crm_session=')` — any request with that substring passes the middleware gate. It's backed by real session validation in `crm-auth.ts` (handlers re-check), so it's not a full bypass — but the middleware layer is cosmetic. Acceptable IF every /api/crm handler validates; verify that's true for all 60 routes (esp. billing/webhook).

## 🟡 MEDIUM
6. **Stripe/webhook auth** — `src/app/api/crm/billing/webhook` exists. Stripe webhooks MUST verify the signature; if it trusts the body without `stripe.webhooks.constructEvent`, it's a critical bypass (fake payments). NOT verified this pass — check `stripe.ts` / webhook route.
7. **Rate limiting** — contact has it (5/15min). Unknown if chat/agent/crm routes do. `/api/agent` has NO rate limit + only the weak key. Brute-forceable if key unknown, but key is default-known.
8. **`.env` is gitignored + no secrets on disk** ✅ (good — opposite of RI's Maps leak). But confirm no secrets were committed historically / in the tar (`Renewably New.tar` in root — check it's not a dumped .env).
9. **CSP allows `unsafe-inline` + `unsafe-eval`** (next.config.ts line 27). Weakens XSS protection. Acceptable for a Next app with inline scripts, but `unsafe-eval` is risky if any user input reaches eval.

## ✅ WHAT'S DONE WELL (don't regress)
- Security headers in CODE (CSP, HSTS, X-Frame-Options, Permissions-Policy, Referrer-Policy) — RI #15 doesn't apply here.
- `.env` gitignored, no plaintext secrets on disk.
- CRM session validation real (not just cookie check).
- Contact form: rate-limited, field-validated, email + DB + activity logging.
- `vitest.config.ts` + `__tests__/` present → SOME test coverage (opposite of RI #35).
